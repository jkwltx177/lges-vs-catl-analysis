# Task 1과 Task 2 병합(Merge) 전략 및 차이점 분석

본 문서는 `main` 브랜치에 반영된 Task 1(자료 조사) 구현체와 `feature/task2-data-processing` 브랜치에 구현된 Task 2(자료 정리) 파이프라인의 차이점을 파악하고, 두 구현체의 장점을 모두 살려 충돌 없이 병합(Merge)하기 위한 전략을 상세히 설명합니다.

---

## 1. 구현 방식의 차이점 분석

### 🅰️ Task 1 구현 방식 (현재 `main` 브랜치)
*   **주요 로직:** `research_nodes.py`의 `company_research_node`에서 검색된 `raw_documents`를 LGES와 CATL 두 기업으로 각각 필터링한 후, LLM 호출 한 번으로 **SWOT(강점/약점/기회/위협/시장) 카테고리까지 직접 분류하여 `company_a`, `company_b` 객체에 저장**합니다.
*   **장점:**
    *   방대한 양의 원문(`raw_documents`)을 기업별로 **사전 필터링(Filtering)**하여 토큰 낭비와 컨텍스트 초과 위험을 줄입니다.
    *   조사 단계에서 즉각적으로 정보를 요약하므로 처리 속도가 매우 빠릅니다.
*   **단점 (위험 요소):**
    *   **과도한 압축(Over-compression):** 긴 원문을 단순한 한 줄짜리 "핵심 사실"로 요약해버려, 분석에 필수적인 구체적 수치, 고유명사, 맥락이 소실됩니다.
    *   **출처(Source) 유실:** LLM이 요약본만 생성하고, 해당 팩트가 어떤 웹사이트/문헌에서 발췌되었는지에 대한 `source` 정보(URL 등)를 매핑하지 않아 **팩트 교차 검증이 불가능**해집니다. (Safe Output 원칙 위배)
    *   시장/포트폴리오 정보가 SWOT과 무분별하게 섞이게 됩니다.

### 🅱️ Task 2 내 방식 (현재 `feature` 브랜치)
*   **주요 로직:** 날것의 데이터(`raw_findings`)를 있는 그대로 받아, `clean_node`에서 맥락과 수치, 출처를 철저히 보존한 채로 팩트를 추출합니다. 이후 `market_node`, `portfolio_node`, `swot_map_node`로 **분리하여 구조화**합니다.
*   **장점:**
    *   **Safe Output 보장:** "수치, 날짜, 기관명 생략 금지", "출처 보존 강제" 등의 룰을 통해 정보의 훼손과 할루시네이션(Hallucination)을 완벽히 방어합니다.
    *   `design.md`에서 요구한 까다로운 세부 평가 기준(캐즘 원인, IRA/CRMA 규제, LRS 모델 등)을 명확하게 타겟팅하여 추출할 수 있습니다.
*   **단점 (위험 요소):**
    *   Task 1에서 정제되지 않은 너무 방대한 텍스트(`raw_documents`)가 통째로 넘어올 경우, LLM의 컨텍스트 한도를 초과하여 중간 정보가 무시되는 현상(Lost in the middle)이 발생할 수 있습니다.

---

## 2. 두 방식의 장점을 모두 살린 합치기(Merge) 전략

**핵심 방향:** Task 1의 **"데이터 필터링/청킹(Chunking) 능력"**과 Task 2의 **"Safe Output 및 세밀한 구조화 능력"**을 결합합니다. 즉, Task 1은 **"안전한 1차 데이터 덩어리(Rich Facts) 생성"**까지만 수행하고, 본격적인 분류와 매핑은 Task 2에게 위임합니다.

### Step 1. Task 1 (`company_research_node`) 프롬프트 수정
Task 1에서 SWOT 분류를 성급하게 수행하여 정보를 뭉개지 못하도록 프롬프트를 변경합니다.
*   **기존:** `"핵심 사실 1을 strengths로 분류하세요..."`
*   **변경 후:** `raw_documents`에서 기업 관련 핵심 정보를 뽑아내되, **"2~3문장 분량의 충분한 맥락 보존", "구체적인 수치/날짜 유지", "각 팩트의 출처(Source URL)를 반드시 매핑"**하여 `RawItem` 리스트로만 변환하게 합니다.
*   *기대 효과:* 방대한 문서를 기업별로 쪼개어 토큰 한도를 우회(Task 1의 장점)하면서도, 정보와 출처의 유실(Task 2의 해결책)을 완벽히 방어할 수 있습니다.

### Step 2. Task 2 (`clean_node`) 역할 최적화
Task 1에서 안전하게 뽑아낸 `company_a(LGES)`, `company_b(CATL)`의 `RawItem` 리스트와 양사 비교 데이터인 `raw_findings`를 인풋으로 받도록 `clean_node`를 수정합니다.
*   **변경 후 로직:** `clean_node`는 이제 무거운 팩트 추출을 할 필요 없이, **중복된 팩트들만 하나로 예쁘게 병합(Deduplication)**하고 구조를 깔끔하게 다듬는 역할로 가벼워집니다.

### Step 3. 하위 노드(`market`, `portfolio`, `swot_map`) 분리 구조 유지
`clean_node`를 거친 깔끔한 팩트 리스트를 바탕으로, 기존에 제가 작성한 Task 2의 `market_node`, `portfolio_node`, `swot_map_node`가 **출처를 유지한 채** 각각의 버킷에 정보를 정확히 담습니다. (이 부분은 수정 없이 기존의 강력한 구조를 그대로 유지합니다.)

---

## 결론
이 방식으로 병합하면, Task 1의 **"효율적인 정보 검색 및 1차 필터링 처리"**의 장점을 온전히 취하면서도, Task 2의 **"정보 유실 없는 Safe Output"**과 **"출처(Source)가 보존된 엄격한 구조화"**를 모두 달성할 수 있습니다. 
이러한 협력적 아키텍처는 최종적으로 Task 4(보고서 작성)에서 가장 신뢰할 수 있고 완벽한 레퍼런스를 갖춘 결과물을 보장합니다.
